# Mini NPU Simulator

MAC(Multiply-Accumulate) 연산을 기반으로 입력 패턴이 십자가(Cross)인지 X인지를 판별하는 시뮬레이터입니다.

## 실행 방법

```bash
python main.py
```

실행하면 모드 선택 화면이 나타납니다.

- **모드 1 (사용자 입력)**: 3x3 필터 2개(A, B)와 패턴을 직접 입력하여 MAC 연산 결과를 확인합니다.
- **모드 2 (data.json 분석)**: data.json에 정의된 5x5, 13x13, 25x25 크기의 필터와 패턴을 자동으로 로드하여 일괄 판정합니다.

data.json은 main.py와 같은 디렉토리에 위치해야 합니다.

### MAC 연산의 전체 흐름 (코드 구조 기준)

MAC 연산은 다음과 같은 흐름으로 동작합니다.

1. **입력 수신**: 2차원 리스트 형태의 pattern과 filter_data를 인자로 받습니다.
2. **크기 결정**: `n = len(pattern)`으로 행렬 크기를 파악합니다.
3. **위치별 곱셈**: 이중 for문(`for i in range(n)`, `for j in range(n)`)으로 모든 (i, j) 위치를 순회하면서 `pattern[i][j] * filter_data[i][j]`를 계산합니다.
4. **누적 합산**: 각 곱셈 결과를 `score` 변수에 누적으로 더합니다 (`score += ...`).
5. **점수 반환**: 모든 위치를 순회한 뒤 최종 score(float)를 반환합니다.

```python
def mac_operation(pattern, filter_data):
    n = len(pattern)
    score = 0.0                              # 누적 합산 변수
    for i in range(n):                       # 행 순회
        for j in range(n):                   # 열 순회
            score += pattern[i][j] * filter_data[i][j]  # 곱셈 + 누적
    return score                             # 최종 점수 반환
```

### MAC 점수와 유사도의 관계

MAC 점수가 높을수록 "더 유사하다"고 판단할 수 있는 이유는 다음과 같습니다.

필터는 특정 패턴(Cross 또는 X)에서 1이 있는 위치를 정의합니다. 입력 패턴에서도 같은 위치에 1이 있으면, 곱셈 결과가 1 x 1 = 1이 되어 점수에 기여합니다. 반대로 입력이 0인 위치는 0 x 1 = 0이 되어 점수에 기여하지 않습니다.

즉, 필터가 "이 위치에 값이 있어야 해"라고 정의한 곳에 실제로 값이 있으면 점수가 올라갑니다. 따라서 입력 패턴이 필터와 모양이 같을수록(=같은 위치에 1이 많을수록) MAC 점수가 높아지고, 이것은 곧 두 패턴이 유사하다는 뜻이 됩니다.

예를 들어 3x3에서:
- 십자가 입력 x 십자가 필터 = 5점 (5개 위치 일치)
- 십자가 입력 x X 필터 = 1점 (중앙 1개만 일치)

점수가 5 vs 1이므로 "십자가에 더 가깝다"고 판정하는 것입니다.

### 라벨 정규화 (normalize_label)

프로그램 내부에서 "표준 라벨" 2가지를 사용합니다: `Cross`, `X`

| 입력 값 | 표준 라벨 |
|---------|----------|
| `+`, `cross`, `Cross` | `Cross` |
| `x`, `X` | `X` |

#### 별도 함수로 분리한 이유

`normalize_label()`을 별도 함수로 분리한 이유는 세 가지입니다.

1. **재사용성**: 라벨 변환이 필요한 곳이 여러 군데입니다. 필터 로드 시 키 정규화, expected 값 정규화, 판정 결과 비교 등에서 모두 같은 변환 규칙을 사용합니다. 함수로 분리하지 않으면 같은 if-else 로직을 여러 번 반복 작성해야 합니다.

2. **유지보수**: 나중에 새로운 라벨(예: 'o'나 'circle')을 추가해야 할 때, 이 함수 하나만 수정하면 프로그램 전체에 반영됩니다. 코드 곳곳에 변환 로직이 흩어져 있으면 하나라도 빠뜨리면 버그가 됩니다.

3. **가독성**: `normalize_label(expected_raw)`이라는 한 줄의 호출만으로 "이 값을 표준 라벨로 바꾸는 것"이라는 의도가 명확하게 드러납니다.

### 패턴 키 → 필터 매칭 로직

data.json의 패턴 키는 `size_{N}_{idx}` 형태입니다 (예: `size_5_1`, `size_13_2`).

매칭 로직은 다음과 같습니다:

```python
# 1. 키를 '_'로 분리
parts = pattern_key.split("_")    # ["size", "5", "1"]

# 2. 두 번째 요소에서 N 추출
n = int(parts[1])                 # 5

# 3. 필터 키 생성
size_key = f"size_{n}"            # "size_5"

# 4. 해당 필터 존재 확인
if size_key not in filters:
    # FAIL 처리
```

이 방식으로 패턴의 크기를 자동으로 파악하고, 대응하는 필터를 찾습니다. 만약 해당 크기의 필터가 없거나, 키 형식이 잘못되었으면 예외를 잡아서 FAIL로 처리하되 프로그램은 중단시키지 않습니다.

### 동점 처리 정책 (epsilon)

부동소수점 연산에서는 수학적으로 같은 값이라도 미세한 오차가 발생할 수 있습니다.
예: `0.1 + 0.1 + ... (8번)` → 기대값 `0.8`, 실제 `0.7999999999999999`

두 점수의 차이가 `1e-9`(0.000000001) 미만이면 동점(`UNDECIDED`)으로 처리합니다.

```python
EPSILON = 1e-9

def compare_scores(score_a, score_b):
    if abs(score_a - score_b) < EPSILON:
        return 'UNDECIDED'
    elif score_a > score_b:
        return 'A'
    else:
        return 'B'
```

#### 모드 1과 모드 2의 동점 처리 차이

동점 처리의 핵심 로직(`compare_scores`)은 동일하지만, 표현 방식과 결과 처리가 다릅니다.

- **모드 1**: 동점 시 `"판정 불가 (|A-B| < 1e-09)"`를 출력합니다. 사용자가 직접 입력한 것이므로 PASS/FAIL 비교 대상이 없고, 단순히 결과를 보여주는 것이 목적이기 때문입니다.

- **모드 2**: 동점 시 `"UNDECIDED"`로 판정합니다. expected 값이 Cross 또는 X이므로, UNDECIDED는 어느 쪽도 아닌 결과가 되어 자동으로 **FAIL**로 집계됩니다. 이는 data.json에 정답이 있으므로 정답과 비교하여 PASS/FAIL을 매길 수 있기 때문입니다.

요구사항 관점에서, 모드 1은 "사용자에게 판정 결과를 보여주는 인터랙티브 도구"이고, 모드 2는 "데이터셋을 자동 검증하는 테스트 도구"이므로 동점의 처리 방식이 다른 것이 맞습니다.

### 시간 측정 경계

시간 측정은 I/O(입출력, 파일 읽기)를 제외하고 **순수 MAC 연산 함수 호출 구간만** 측정합니다.

```python
def measure_mac_time(pattern, filter_data, repeat=10):
    times = []
    for _ in range(repeat):
        start = time.perf_counter()       # ← 측정 시작 (연산 직전)
        mac_operation(pattern, filter_data)  # ← 순수 연산만 포함
        end = time.perf_counter()         # ← 측정 종료 (연산 직후)
        times.append((end - start) * 1000)
    avg = sum(times) / len(times)
    return avg
```

- `time.perf_counter()`를 사용한 이유: `time.time()`보다 정밀도가 높고(마이크로초 단위), CPU 시간이 아닌 벽시계 시간(wall-clock time)을 측정합니다.
- JSON 파일 읽기, 콘솔 출력, 패턴 로드 등의 I/O 작업은 측정 구간에 포함되지 않습니다.
- 10회 반복 후 평균을 내어 1회 측정의 노이즈(OS 스케줄링, 캐시 상태 등)를 줄입니다.

### 보너스 1: 메모리 최적화 (1D 배열)

2차원 배열을 1차원 배열로 펼쳐서(flatten) MAC 연산을 수행합니다. 2D 배열은 `array[i][j]`처럼 이중 인덱싱이 필요하지만, 1D 배열은 `array[k]`로 한 번만 접근합니다.

```python
def flatten_2d(matrix):
    result = []
    for row in matrix:
        for val in row:
            result.append(val)
    return result

def mac_operation_1d(pattern_1d, filter_1d):
    score = 0.0
    for i in range(len(pattern_1d)):
        score += pattern_1d[i] * filter_1d[i]
    return score
```

프로그램 실행 시 2D와 1D 방식의 연산 시간을 비교하는 표가 출력됩니다.

### 보너스 2: 패턴 생성기

크기 N을 입력하면 N x N 십자가 패턴과 X 패턴을 자동으로 생성합니다. 모드 1에서 패턴 생성기 사용 여부를 선택할 수 있고, 성능 분석에서도 3x3 크기 데이터를 생성하는 데 재활용됩니다.

- **Cross 패턴**: 가운데 행 전체와 가운데 열 전체를 1로 설정
- **X 패턴**: 주대각선(좌상→우하)과 부대각선(우상→좌하)을 1로 설정

### FAIL 케이스 분석

**size_5_4: UNDECIDED (동점 규칙에 의한 FAIL)**

이 케이스는 5x5 패턴의 모든 값이 0.1로 채워져 있습니다. Cross 필터와 X 필터 모두 5x5 크기에서 9개의 1을 가지고 있으므로(Cross: 가운데 행 5개 + 가운데 열 5개 - 중앙 겹침 1개 = 9개, X: 주대각선 5개 + 부대각선 5개 - 중앙 겹침 1개 = 9개), 두 필터의 MAC 점수가 동일하게 계산됩니다.

- Cross 점수: 0.1 x 9 = 0.9 (실제 부동소수점 값: 0.8999999999999999)
- X 점수: 0.1 x 9 = 0.9 (실제 부동소수점 값: 0.8999999999999999)

두 점수의 차이(|0.8999999999999999 - 0.8999999999999999| = 0.0)가 epsilon(1e-9) 미만이므로 UNDECIDED로 판정되었고, expected가 "X"였기 때문에 FAIL이 되었습니다.

이 실패는 프로그램 로직의 버그가 아니라, 입력 데이터 자체가 두 패턴 사이에서 구분이 불가능한 경우입니다. 모든 값이 동일한 패턴은 Cross도 X도 아니기 때문에 UNDECIDED가 올바른 판정입니다.

### 실패 원인의 분류

실패 원인을 세 가지로 분류할 수 있습니다.

1. **데이터/스키마 문제**: JSON 키 이름이 예상과 다르거나, 필터와 패턴의 크기가 맞지 않는 경우. 프로그램에서는 키에서 크기를 추출하고 필터와 대조하여 검증하기 때문에, 이 문제가 있으면 바로 FAIL로 처리하고 다음 케이스로 넘어갑니다(프로그램 중단 없음).

2. **로직 문제**: MAC 연산 자체가 잘못되었거나, 라벨 정규화가 누락된 경우. 이번 구현에서는 `+` → `Cross`, `x` → `X`로 정규화하고, 비교 시 항상 표준 라벨을 사용하기 때문에 로직 문제로 인한 실패는 없었습니다.

3. **수치 비교 문제 (부동소수점 오차)**: 부동소수점 연산의 특성상 두 점수가 수학적으로 같더라도 다르게 계산될 수 있습니다. epsilon 기반 비교 정책으로 이 문제를 처리하였고, size_5_4 케이스처럼 실제로 두 점수가 거의 같은 경우에는 UNDECIDED로 올바르게 처리됩니다.

이번 테스트에서 9개가 PASS인 이유는 세 가지입니다:
- 라벨 정규화를 정확히 적용하여 `+`와 `Cross`, `x`와 `X`를 올바르게 매핑하였습니다.
- epsilon 비교 정책으로 부동소수점 오차가 판정에 영향을 주지 않도록 하였습니다.
- 스키마 검증(크기 일치, 필터 존재 여부)을 통해 구조적 문제를 사전에 감지하였습니다.
유일한 FAIL(size_5_4)은 입력 자체가 판별 불가능한 데이터였으며, UNDECIDED 판정은 정확합니다.

### 성능 분석 결과

아래는 각 크기별 MAC 연산 시간을 10회 반복 측정한 평균값입니다.

| 크기 | 평균 시간(ms) | 연산 횟수(N^2) |
|------|-------------|-------------|
| 3x3 | 약 0.001 | 9 |
| 5x5 | 약 0.002 | 25 |
| 13x13 | 약 0.010 | 169 |
| 25x25 | 약 0.033 | 625 |

(실행 환경에 따라 수치는 달라질 수 있습니다)

### 시간 복잡도 분석: O(N^2) 근거

MAC 연산은 N x N 크기의 두 행렬에서 같은 위치의 값을 곱하고 합산하는 연산입니다. 코드의 이중 for문을 보면, 바깥 루프가 N번, 안쪽 루프가 N번 반복되므로 총 곱셈 횟수는 정확히 N^2입니다.

```python
for i in range(n):       # N번
    for j in range(n):   # N번
        score += pattern[i][j] * filter_data[i][j]  # 1번 곱셈 + 1번 덧셈
```

따라서 MAC 연산의 시간 복잡도는 **O(N^2)** 입니다.

실제 측정 결과와 비교해 보면 이 관계가 맞는지 확인할 수 있습니다. 연산 횟수가 약 k배 늘어나면, 시간도 대략 k배에 가깝게 늘어나야 합니다.

- 3x3 -> 5x5: 연산 횟수 9 -> 25 (약 2.8배), 시간도 약 2~3배 증가
- 5x5 -> 13x13: 연산 횟수 25 -> 169 (약 6.8배), 시간도 약 5~6배 증가
- 13x13 -> 25x25: 연산 횟수 169 -> 625 (약 3.7배), 시간도 약 3~4배 증가

측정된 시간 증가율이 연산 횟수 증가율과 비례하는 것을 확인할 수 있으며, 이는 시간 복잡도가 O(N^2)임을 뒷받침합니다. 작은 크기에서는 측정 오차의 비율이 상대적으로 크기 때문에 정확한 비례 관계가 아닐 수 있지만, 크기가 커질수록 O(N^2) 경향이 더 명확하게 나타납니다.

### 2D vs 1D 메모리 최적화 비교

2D 배열을 1D 배열로 변환하여 MAC 연산을 수행하면 인덱싱 오버헤드가 줄어들 수 있습니다. 2D에서는 `array[i][j]`로 두 번의 리스트 접근이 필요하지만, 1D에서는 `array[k]`로 한 번만 접근합니다.

| 크기 | 2D 시간(ms) | 1D 시간(ms) | 차이 |
|------|-----------|-----------|------|
| 3x3 | 0.001 | 0.001 | 약 0.000 |
| 5x5 | 0.002 | 0.001 | 약 +0.001 |
| 13x13 | 0.010 | 0.007 | 약 +0.003 |
| 25x25 | 0.033 | 0.025 | 약 +0.008 |

측정 결과 1D 방식이 근소하게 빠른 경향을 보였으며, 크기가 커질수록 그 차이가 약간 더 벌어졌습니다. 실제 NPU에서는 메모리 접근 패턴이 성능에 큰 영향을 미치기 때문에, 1D 배열로 변환하여 연속된 메모리에 접근하는 것이 캐시 효율을 높이는 중요한 최적화 기법이 됩니다.

### 부동소수점 오차가 발생하는 이유와 epsilon이 필요한 이유

컴퓨터는 소수를 2진법(IEEE 754)으로 저장합니다. 0.1 같은 십진 소수는 2진법으로 정확하게 표현할 수 없어서, 내부적으로 0.1000000000000000055511151231257827021181583404541015625 같은 근사값으로 저장됩니다.

이 때문에 0.1을 9번 더하면 정확히 0.9가 아니라 0.8999999999999999가 될 수 있습니다. 만약 단순히 `==`로 비교하면, 수학적으로 같은 두 점수가 "다르다"고 판정될 수 있습니다.

epsilon(허용오차) 기반 비교는 이 문제를 해결합니다. 두 점수의 차이가 epsilon(1e-9) 미만이면 "사실상 같다"고 간주하여 UNDECIDED로 처리합니다. 이렇게 하면 부동소수점 오차로 인한 잘못된 판정을 방지할 수 있습니다.

### 새로운 라벨 추가 시 확장 방안

만약 data.json에 새로운 라벨(예: 'o' 또는 'circle')이 추가된다면, 다음 순서로 확장합니다:

1. **정규화 함수 수정**: `normalize_label()`에 `'o'`, `'circle'` -> `'Circle'` 매핑을 추가합니다.
2. **필터 추가**: data.json의 filters에 circle 필터(원형 패턴의 2D 배열)를 추가합니다.
3. **판정 로직 확장**: 현재는 2개 필터 비교이지만, 3개 이상이면 "가장 높은 점수의 필터"를 선택하도록 변경하고, 상위 2개 점수가 동점이면 UNDECIDED로 처리합니다.
4. **출력 형식 수정**: 결과 출력에 Circle 점수를 추가합니다.
5. **테스트 데이터 추가**: data.json의 patterns에 circle 패턴과 expected: "o" 케이스를 추가합니다.

normalize_label()을 별도 함수로 분리해 둔 덕분에, 1단계에서 함수 하나만 수정하면 프로그램 전체에서 새 라벨이 올바르게 인식됩니다.

### 1000x1000 패턴 처리 시 예상 문제와 최적화

**시간 문제**: 1000x1000은 연산 횟수가 1,000,000(백만)번입니다. 현재 25x25(625번)에서 약 0.03ms이므로, 단순 비례하면 약 48ms가 됩니다. Python의 순수 for문은 느리기 때문에, 실제로는 더 오래 걸릴 수 있습니다. 10회 반복 측정까지 하면 약 0.5초인데, 이 정도는 견딜 수 있지만 실시간 처리에는 부족합니다.

**메모리 문제**: 1000x1000 2D 리스트는 약 8MB(float 8바이트 x 100만)를 차지합니다. Python 리스트는 객체 오버헤드가 있어 실제로는 더 많은 메모리를 사용합니다. 여러 필터와 패턴을 동시에 메모리에 올리면 문제가 될 수 있습니다.

**우선 적용할 최적화**:
1. 1D 배열 변환으로 인덱싱 오버헤드 줄이기 (이미 구현됨)
2. NumPy 같은 벡터화 라이브러리 사용 (과제 제약에서는 금지)
3. C 확장 모듈이나 ctypes를 통한 네이티브 연산
4. 멀티스레딩으로 필터별 MAC 연산 병렬 수행

### CPU 직렬 처리 vs NPU 병렬 처리: 병목 분석

이 시뮬레이터에서 구현한 MAC 연산은 CPU의 직렬 처리 방식입니다. for문이 (i, j)를 하나씩 순서대로 방문하면서 곱셈과 덧셈을 반복합니다.

**CPU의 병목**: 1000x1000 패턴에서 백만 번의 곱셈을 한 번에 하나씩 처리해야 합니다. CPU 코어가 아무리 빨라도, 순차적으로 100만 번 반복해야 하므로 코어 하나의 처리 속도에 제약을 받습니다.

**NPU가 해결하는 방식**: NPU는 수천 개의 MAC 유닛을 내장하고 있어서, 예를 들어 1000개의 곱셈을 동시에 수행할 수 있습니다. 1000x1000 = 100만 번의 연산을 1000개씩 묶어서 병렬 처리하면, 이론적으로 1000번만 반복하면 됩니다. 즉 처리 시간이 1/1000로 줄어듭니다.

**실제 병목 지점**: 대규모 MAC에서 가장 큰 병목은 "연산 자체"보다 "데이터를 연산기로 전달하는 메모리 대역폭"에 있는 경우가 많습니다. 아무리 연산기가 많아도, 데이터를 메모리에서 읽어오는 속도가 느리면 연산기가 놀게 됩니다. 이것을 "메모리 바운드(memory-bound)"라고 하며, NPU 설계에서 메모리 접근 최적화(데이터 재사용, 타일링 등)가 중요한 이유이기도 합니다.
