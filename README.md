# Mini NPU Simulator

MAC(Multiply-Accumulate) 연산을 기반으로 입력 패턴이 십자가(Cross)인지 X인지를 판별하는 시뮬레이터입니다.

## 실행 방법

```bash
python main.py
```

실행하면 모드 선택 화면이 나타납니다.

- **모드 1 (사용자 입력)**: 3x3 필터 2개(A, B)와 패턴을 직접 입력하여 MAC 연산 결과를 확인합니다.
- **모드 2 (data.json 분석)**: data.json에 정의된 5x5, 13x13, 25x25 크기의 필터와 패턴을 자동으로 로드하여 일괄 판정합니다.

data.json은 main.py와 같은 디렉토리에 위치해야 합니다.

### 모드 1 예시

```
=== Mini NPU Simulator ===

[모드 선택]
1. 사용자 입력 (3x3)

판정: A
```

## 구현 요약

### 라벨 정규화

data.json의 expected 값과 필터 키는 다양한 형태로 들어올 수 있어서, 프로그램 내부에서 통일된 표준 라벨로 변환하여 사용했습니다.

| 입력 값 | 표준 라벨 |
|---------|----------|
| `+`, `cross`, `Cross` | `Cross` |
| `x`, `X` | `X` |

`normalize_label()` 함수에서 입력을 소문자로 변환한 뒤 매핑하는 방식으로 구현했습니다. 이렇게 하면 JSON 데이터의 표기 방식이 달라도 비교 시 문제가 생기지 않습니다.

### MAC 연산

MAC(Multiply-Accumulate) 연산은 두 개의 2차원 배열에서 같은 위치의 값끼리 곱한 뒤 전부 더하는 연산입니다. 외부 라이브러리 없이 이중 for문으로 직접 구현했습니다.

```python
def mac_operation(pattern, filter_data):
    n = len(pattern)
    score = 0.0
    for i in range(n):
        for j in range(n):
            score += pattern[i][j] * filter_data[i][j]
    return score
```

입력 패턴에 대해 Cross 필터와 X 필터 각각의 MAC 점수를 구한 뒤, 점수가 더 높은 쪽으로 판정합니다.

### 동점 처리 정책 (epsilon)

부동소수점 연산에서는 수학적으로 같은 값이라도 미세한 오차가 발생할 수 있습니다. 예를 들어 `0.1 + 0.1 + ... + 0.1`을 8번 더하면 정확히 `0.8`이 아니라 `0.7999999999999999` 같은 값이 나올 수 있습니다.

그래서 두 점수의 차이가 `1e-9`(0.000000001) 미만이면 동점(`UNDECIDED`)으로 처리하도록 했습니다.

```python
EPSILON = 1e-9

def compare_scores(score_a, score_b):
    if abs(score_a - score_b) < EPSILON:
        return 'UNDECIDED'
    elif score_a > score_b:
        return 'A'
    else:
        return 'B'
```

### 보너스 1: 메모리 최적화 (1D 배열)

2차원 배열을 1차원 배열로 펼쳐서(flatten) MAC 연산을 수행하는 방식도 구현했습니다. 2D 배열은 `array[i][j]`처럼 이중 인덱싱이 필요하지만, 1D 배열은 `array[k]`로 한 번만 접근하면 됩니다.

```python
def flatten_2d(matrix):
    result = []
    for row in matrix:
        for val in row:
            result.append(val)
    return result

def mac_operation_1d(pattern_1d, filter_1d):
    score = 0.0
    for i in range(len(pattern_1d)):
        score += pattern_1d[i] * filter_1d[i]
    return score
```

프로그램 실행 시 2D와 1D 방식의 연산 시간을 비교하는 표가 출력됩니다.

### 보너스 2: 패턴 생성기

크기 N을 입력하면 N x N 십자가 패턴과 X 패턴을 자동으로 생성합니다. 모드 1에서 패턴 생성기 사용 여부를 선택할 수 있습니다.

- **Cross 패턴**: 가운데 행 전체와 가운데 열 전체를 1로 설정
- **X 패턴**: 주대각선(좌상→우하)과 부대각선(우상→좌하)을 1로 설정

## 결과 리포트

### 테스트 결과 요약

data.json에 10개의 테스트 케이스를 준비하여 실행한 결과는 다음과 같습니다.

| 항목 | 값 |
|------|-----|
| 총 테스트 | 10개 |
| 통과 (PASS) | 9개 |
| 실패 (FAIL) | 1개 |

### FAIL 케이스 분석

**size_5_4: UNDECIDED (동점 규칙에 의한 FAIL)**

이 케이스는 5x5 패턴의 모든 값이 `0.1`로 채워져 있습니다. Cross 필터와 X 필터 모두 9개의 1을 가지고 있으므로, 두 필터의 MAC 점수가 동일하게 계산됩니다.

- Cross 점수: 0.1 * 9 = 0.9 (정확히는 `0.8999999999999999`)
- X 점수: 0.1 * 9 = 0.9 (정확히는 `0.8999999999999999`)

두 점수의 차이가 `1e-9` 미만이므로 UNDECIDED로 판정되었고, expected가 `X`였기 때문에 FAIL이 되었습니다.

이 실패는 프로그램 로직의 버그가 아니라, 입력 데이터 자체가 두 패턴 사이에서 구분이 불가능한 경우입니다. 실제로 모든 값이 같은 패턴은 Cross도 X도 아니기 때문에 UNDECIDED가 올바른 판정입니다.

### 실패 원인의 분류

실패 원인을 세 가지로 분류할 수 있습니다.

1. **데이터/스키마 문제**: JSON 키 이름이 예상과 다르거나, 필터와 패턴의 크기가 맞지 않는 경우. 프로그램에서는 키에서 크기를 추출하고 필터와 대조하여 검증하기 때문에, 이 문제가 있으면 바로 FAIL로 처리됩니다.

2. **로직 문제**: MAC 연산 자체가 잘못되었거나, 라벨 정규화가 누락된 경우. 이번 구현에서는 `+` → `Cross`, `x` → `X`로 정규화하고, 비교 시 항상 표준 라벨을 사용하기 때문에 로직 문제로 인한 실패는 없었습니다.

3. **수치 비교 문제 (부동소수점 오차)**: 부동소수점 연산의 특성상 두 점수가 수학적으로 같더라도 다르게 계산될 수 있습니다. epsilon 기반 비교 정책으로 이 문제를 처리하였고, size_5_4 케이스처럼 실제로 두 점수가 거의 같은 경우에는 UNDECIDED로 올바르게 처리됩니다.

이번 테스트에서 9개가 PASS인 이유는 라벨 정규화를 정확히 적용했고(`+` → `Cross`, `x` → `X`), epsilon 비교 정책으로 부동소수점 오차가 판정에 영향을 주지 않도록 했기 때문입니다. 유일한 FAIL(size_5_4)은 입력 자체가 판별 불가능한 데이터였습니다.

### 성능 분석 결과

아래는 각 크기별 MAC 연산 시간을 10회 반복 측정한 평균값입니다.

| 크기 | 평균 시간(ms) | 연산 횟수(N^2) |
|------|-------------|-------------|
| 3x3 | 약 0.001 | 9 |
| 5x5 | 약 0.002 | 25 |
| 13x13 | 약 0.010 | 169 |
| 25x25 | 약 0.033 | 625 |

(실행 환경에 따라 수치는 달라질 수 있습니다)

### 시간 복잡도 분석: O(N^2) 근거

MAC 연산은 N x N 크기의 두 행렬에서 같은 위치의 값을 곱하고 합산하는 연산입니다. 코드의 이중 for문을 보면, 바깥 루프가 N번, 안쪽 루프가 N번 반복되므로 총 곱셈 횟수는 정확히 N^2입니다.

```python
for i in range(n):       # N번
    for j in range(n):   # N번
        score += pattern[i][j] * filter_data[i][j]  # 1번 곱셈 + 1번 덧셈
```

따라서 MAC 연산의 시간 복잡도는 **O(N^2)** 입니다.

실제 측정 결과와 비교해 보면 이 관계가 맞는지 확인할 수 있습니다. 연산 횟수가 약 k배 늘어나면, 시간도 대략 k배에 가깝게 늘어나야 합니다.

- 3x3 → 5x5: 연산 횟수 9 → 25 (약 2.8배), 시간도 약 2~3배 증가
- 5x5 → 13x13: 연산 횟수 25 → 169 (약 6.8배), 시간도 약 5~6배 증가
- 13x13 → 25x25: 연산 횟수 169 → 625 (약 3.7배), 시간도 약 3~4배 증가

측정된 시간 증가율이 연산 횟수 증가율과 비례하는 것을 확인할 수 있으며, 이는 시간 복잡도가 O(N^2)임을 뒷받침합니다. 작은 크기에서는 측정 오차의 비율이 상대적으로 크기 때문에 정확한 비례 관계가 아닐 수 있지만, 크기가 커질수록 O(N^2) 경향이 더 명확하게 나타납니다.

### 2D vs 1D 메모리 최적화 비교

2D 배열을 1D 배열로 변환하여 MAC 연산을 수행하면 인덱싱 오버헤드가 줄어들 수 있습니다. 2D에서는 `array[i][j]`로 두 번의 리스트 접근이 필요하지만, 1D에서는 `array[k]`로 한 번만 접근합니다.

측정 결과 1D 방식이 근소하게 빠른 경향을 보였으며, 크기가 커질수록 그 차이가 약간 더 벌어졌습니다. 다만 Python 인터프리터의 특성상 C/C++에서처럼 극적인 차이는 나지 않았습니다. 실제 NPU에서는 메모리 접근 패턴이 성능에 큰 영향을 미치기 때문에, 1D 배열로 변환하여 연속된 메모리에 접근하는 것이 캐시 효율을 높이는 중요한 최적화 기법이 됩니다.
